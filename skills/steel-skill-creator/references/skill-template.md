# Generated skill template

Use this scaffold when authoring the SKILL.md for the skill you're generating. Fill in every section; delete anything you don't need rather than leaving placeholder text. Aim for 60–120 lines of body. The trace is the source code; the SKILL.md is the abstract — and the abstract leans on the `steel-browser` skill for the actual primitives.

## Generated skills depend on `steel-browser`

Every generated skill assumes the executor has the `steel` CLI installed and the `steel-browser` skill loaded. The executor invokes `steel-browser` to drive the actual browser; the generated skill describes *what* to do in plain intent, not *how* to do it command by command.

This is the right shape because:

- The steel CLI and its primitive names (`snapshot`, `click @eN`, `fill`, `wait`, `batch`, `eval`, `stop`) live in one place. Changes there flow to all generated skills automatically.
- A generated skill stays terse and readable — what the *abstract* should be.
- Onboarding for a new user is one curl command: `curl -fsS https://setup.steel.dev | sh`. After that, anything generated by `steel-skill-creator` runs.
- The executor's intelligence is the runtime. Telling it "click the *Where from?* combobox and fill it with `{origin}`" is enough; it knows from `steel-browser` how to express that as a `snapshot → find @eN → click → fill → wait` sequence.

Every generated skill must include a `## Prerequisites` section that states this dependency and points at the install command. See the worked example below for the exact wording.

## Frontmatter

```yaml
---
name: <kebab-case-name>
description: <one sentence on what the skill does, followed by a sentence on when to trigger it. Be a little pushy on triggering — Claude tends to under-trigger skills. Include the specific phrases/contexts a user would say when this skill applies. End with "Requires the `steel-browser` skill (install via `curl -fsS https://setup.steel.dev | sh`)." so future Claudes know the dependency.>
---
```

## Body structure

```markdown
# <Skill display name>

<One short paragraph on what this skill is for. Lead with the outcome.>

## Prerequisites

This skill drives a real browser via Steel. If you don't already have Steel set up:

```bash
curl -fsS https://setup.steel.dev | sh
```

That installs the `steel` CLI and the `steel-browser` skill into `~/.claude/skills/`. Then export `STEEL_API_KEY` (get one at app.steel.dev/settings/api-keys). If `steel-browser` wasn't loaded when Claude Code started, restart so it picks up the new skill, then invoke this skill again.

This skill assumes `steel-browser` is available and uses its primitives.

## Inputs

- `<param_name>` — <what it is, format expected, example value>
- `<param_name>` — <...>

<If the skill has additional preconditions the executor cannot satisfy themselves — Steel credentials in the vault for a specific origin, a profile that must exist, a paid Steel plan feature — list them in a "Site-specific preconditions" subsection.>

## Steps

1. **Start a Steel session.** <Describe the configuration the session needs: stealth on if the site is bot-heavy, proxy region if geo-sensitive, credentials/profile if logged-in state is needed.>

2. **<Phase name>.** <Terse intent prose. Name elements by accessibleName. State waits explicitly.>

...

N. **Stop the session.**

## Output

<What the skill returns. Be concrete: a JSON shape, a file path, a price. Include an example.>

## Notes & caveats

<Real-world failure modes you observed during the two recording runs. This is operator wisdom — don't leave it out.>
```

## Style guidance for the body

**Lean on `steel-browser` primitives.** The executor knows them. You write "click the *Where from?* combobox and fill it with `{origin}`"; the executor turns that into the snapshot-find-click-fill sequence. Don't expand it for them — it's noise.

**Refer to elements by their accessibleName, in italics.** "Click the *Done* button" beats "click the button at the top of the date picker". The accessibleName comes straight from the trace and is what the executor will look for during snapshot.

**State waits as part of the step that needs them.** "Wait for the suggestion list to appear, then pick the first option whose accessibleName ends with `(IATA)`." Not a separate "wait for networkidle" step floating in the middle.

**Imperative voice.** "Click X." Not "The skill should click X."

**Templated URLs beat click sequences.** If a user input ends up in the URL (`?from=SPU&to=FCO`), navigate to the templated URL directly. Skip the intermediate clicks. Every click skipped is one less brittle DOM dependency.

**Use `eval` patterns for bulk reads or stable-attribute clicks.** If the step is "read the aria-label of every gridcell and parse them", describe that intent and the executor knows to reach for `steel browser eval`. Same for "click the gridcell whose aria-label is X" — describe the intent, not the eval syntax.

**Lead with intent, not mechanism.** "Apply the nonstop filter" beats "click the *Stops* filter button, then click the *Nonstop only* radio, then click *Close dialog*". You can mention the elements involved as a sub-bullet if needed, but the top-line is intent.

**Don't include the original trace's parameter values.** If trace #1 was SPU→FCO and trace #2 was JFK→LAX, the generated skill takes `origin` and `destination`, not "SPU" or "JFK". Hardcoding either is a tell that you didn't parameterize.

**Don't include trace IDs, session IDs from your recording runs, or timestamps.** Those are compilation artifacts; they don't belong in the reusable skill.

## Worked example (excerpt)

This is roughly what the steps look like for the `cheapest-direct-flight` skill. Note the intent-first phrasing, italicized accessibleNames, named primitives without expanded commands:

```markdown
## Steps

1. **Start a stealth Steel session.** Google triggers bot challenges intermittently, so default stealth on. 10-minute timeout is enough for a full flow.

2. **Open Google Flights with USD pinned.** Navigate to `https://www.google.com/travel/flights?hl=en&curr=USD`, wait for `networkidle`.

3. **Set origin.** Click the *Where from?* combobox, fill it with `{origin}`, wait for the suggestion list, pick the first option whose accessibleName ends with `(IATA)`. If the field shows stale text, send `Control+A` then `Backspace` before filling.

4. **Set destination.** Same pattern with the *Where to?* combobox and `{destination}`.

5. **Set departure date.** Open the date picker by clicking the *Departure* textbox. Click the gridcell whose aria-label matches `{depart_date}` formatted as "Weekday, Month D, YYYY". Click any later gridcell as a throwaway return (the Date grid in step 8 overrides it). Click the *Done* button. If refs churn between snapshots in the picker, fall back to clicking by aria-label via `eval`.

6. **Run the search.** Click *Search*, wait until the URL contains `/travel/flights/search` and the text "round trip" appears.

7. **Apply the Nonstop filter.** Open the *Stops* filter, pick *Nonstop only*, close the dialog, wait for `networkidle`.

8. **Open the Date grid.** Click *Date grid*, wait for `networkidle`.

9. **Read all cells and pick the cheapest.** Read every gridcell's aria-label in one `eval` — each is shaped like `"$NNN, MMM D to MMM D"`. Parse into `(price, depart, return)`, filter to the requested `flex_days` and `min/max_nights` window, pick the minimum. If zero match, wait 3s and re-read once (the grid lags the filter state); still zero, return `null`.

10. **Confirm the pick.** Click the winning gridcell by aria-label, click *OK*, wait for `networkidle`. If "Oops, something went wrong" renders, click *Reload* once; if it fails twice, abort.

11. **Read the first flight.** Find the *Top departing flights* heading (fall back to *Other departing flights*), read the first listed flight's text. Parse out airline, times, price.

12. **Stop the session.**
```

That's the whole shape — 12 numbered steps, each one a sentence or two of intent, no expanded CLI commands. The executor reads this and runs it through `steel-browser`.
