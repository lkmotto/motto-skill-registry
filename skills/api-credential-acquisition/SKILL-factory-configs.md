---name: api-credential-acquisitiondescription: Reuse the standardized auth runner to acquire foreign API tokens across providers (with Bitwarden credential sources and Doppler writes) instead of rebuilding flows from scratch.---# API Credential AcquisitionUse this skill when the user asks to collect, rotate, or persist external API tokens.## Canonical implementation- Runner: `C:\Users\lkmot\tools\auth_runner.py`- Recipes:  - `C:\Users\lkmot\tools\auth_runner_recipes.sample.json`  - `C:\Users\lkmot\tools\auth_runner_recipes.live-test.json`## Standard recipe contract- `credentials.username|password|totp|email`: typically `bw:item:<item>:<field>`- `acquisition.provider` + `acquisition.mode`: provider-specific token acquisition strategy- `acquisition.params`: provider-specific inputs- `doppler.value`: use `acquisition:value` for standardized output wiring## Runtime conventions1. Resolve credentials from Bitwarden first.2. Run provider acquisition via dispatcher (`provider/mode`).3. Store standardized result in runtime keys:   - `acquisition.value`   - `acquisition.path`   - `acquisition.meta.*`4. Persist to Doppler.5. Return masked proof (hash/length), never raw token.## Service-to-service onboarding guardrail (minimize plumbing time)Before any live acquisition, run this one-time onboarding sequence for a provider:1. Verify connector auth state (`connected` and tool-authorized, not just `connected`).2. If provider uses OAuth tickets/scopes, run an auth-probe call first.3. Cache provider bootstrap source (e.g., seed token path) in recipe params.4. Require a deterministic acquisition mode (`provider` + `mode`) before execution.5. Fail fast with explicit blocker reason if integration state is incomplete.This prevents rebuilding auth plumbing every session and keeps runs focused on token acquisition.## Validation command```powershellpython C:\Users\lkmot\tools\auth_runner.py --recipes C:\Users\lkmot\tools\auth_runner_recipes.live-test.json --service <service-name> --live```
